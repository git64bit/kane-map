# Real Data Plan

Status: planning document.

This document defines the next phase: moving from synthetic demo chunks toward real Kane County geometry without damaging the offline-first architecture.

## Current state

The app currently uses synthetic data chunks and a pure offline Canvas renderer.

The core fieldwork workflow works:

- select buildings
- enter visible designators
- auto-count units
- save observations locally
- edit/delete records
- filter/review coverage
- plan visits
- export/import backups
- export CSV/TXT reports

The next risk is real-data intake, not UI mechanics.

## Real-data principle

```text
Heavy processing happens outside the browser.
Field use happens inside the browser.
The browser consumes prepared static data.
```

## Preferred production data flow

```text
Public source data
  -> Debian processing node
  -> Python venv processing scripts
  -> normalized geometry
  -> Kane-grid assignment
  -> static chunks
  -> local Kane-Map data folder
  -> offline browser rendering
```

This is preferred, not mandatory.

## Why not import directly into the browser?

The browser should not be responsible for:

- large source downloads
- coordinate-system conversion
- shapefile parsing
- projection decisions
- source metadata review
- topology repair
- geometry simplification
- chunk generation
- provenance logs

Those tasks belong to a repeatable processing environment.

## First real-data target

Do not start with every layer.

Recommended first target:

```text
1. County boundary or local bounding envelope
2. Road centerlines or simplified road geometry
3. Water polygons
4. Forest/green orientation polygons
5. Address points or building candidates
6. Building footprints only after source review
```

Building footprints are operationally important, but they should not be rushed because they become the anchor for field observations.

## Demo replacement strategy

Do not delete synthetic chunks immediately.

Instead add a mode flag or data catalog:

```text
dataMode: demo | production
```

Suggested catalog:

```text
data/catalog.json
```

For direct-open mode:

```text
data/catalog.js
```

Then the app can load either:

```text
src/data/chunks/demo/*.js
```

or:

```text
data/chunks/kane-v001/*.js
```

## Production chunks

Suggested production chunk files:

```text
data/chunks/kane-v001/N14-E08.js
data/chunks/kane-v001/N14-E09.js
data/chunks/kane-v001/N15-E08.js
```

Each chunk should include layer groups similar to the current demo model:

```text
buildings
roads
water
forests
gridCells
labels
```

## Data catalog fields

A future catalog should include:

```text
data_version
grid_version
build_date
processing_environment
source_inventory
chunk_count
feature_count_by_layer
coordinate_model
license_notes
validation_report
```

## Source approval sequence

For each candidate source:

```text
candidate
reviewed
approved_for_processing
processed
validated
approved_for_app
retired
```

Do not skip from candidate to app import.

## Minimum validation before app use

Before production data replaces or supplements demo data:

- app opens offline
- chunks load without network
- pan/zoom/rotate remain responsive
- search still works
- building click selection still works
- field observations attach to stable building IDs
- export/import remains compatible
- source catalog is present
- demo mode remains available for testing

## Open questions

These should be resolved before first production import:

```text

What coordinate system should production chunks use?
Should production building IDs be generated by grid+sequence or source ID?
Should address points appear as separate POIs or attach to building candidates?
Should parcels be imported at all, hidden by default, or omitted from the first import?

```
